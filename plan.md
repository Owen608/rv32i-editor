# RV32I 在线编辑器 — 技术实现方案

## 一、目标与范围

- **指令集**：RV32I + 整数乘除法扩展（M），不含浮点。
- **代码层级**：C 源码 → 带伪指令汇编 → 不带伪指令汇编 → 机器码；支持转换过程展示与跟踪。
- **执行**：连续执行、断点、单步；寄存器、指令存储器、数据存储器查看与初始化。
- **环境**：Windows，安装步骤少、实现尽量简单。

---

## 二、整体架构（尽量简化）

采用 **本地单页前端 + 轻量本地后端**，一次配置即可使用。

```
[ 浏览器 ]  ←→  [ 本地后端 API ]  ←→  [ RISC-V 工具链 ]
    │                    │
    ├─ 编辑器 / 多视图
    ├─ 汇编/机器码展示
    └─ JS 模拟器（单步/断点/连续）
```

- **前端**：纯静态 HTML + CSS + JavaScript（或极简 Vue/无构建），负责编辑、展示、模拟器 UI。
- **后端**：单一进程（Node 或 Python 二选一），只做「调工具链 + 读结果」，无数据库、无复杂逻辑。
- **工具链**：官方 RISC-V GNU 工具链（gcc + as + objdump），Windows 下一键解压 + 配 PATH 即可。

这样 C 编译、伪指令展开、机器码生成都交给成熟工具链，我们只做「调用 + 解析」和「前端模拟器」。

---

## 三、技术选型

| 模块         | 方案                     | 说明 |
|--------------|--------------------------|------|
| 前端         | 单页 HTML + 原生 JS 或 轻量 Vue | 无构建或仅需一条 npm 命令，便于在 Windows 快速打开、调试。 |
| 后端         | **Node.js** 或 **Python 3** 二选一 | 只起一个 HTTP 服务，接收源码，调 gcc/as/objdump，返回 JSON（汇编列表、机器码、错误信息）。 |
| C → 汇编     | `riscv-none-elf-gcc -S`  | 工具链自带，稳定。 |
| 汇编 → 机器码 | `riscv-none-elf-as` + `objdump -d` / `objcopy -O binary` | 伪指令由 GAS 展开；objdump 得到反汇编与机器码。 |
| 模拟器       | **前端 JavaScript 实现** | 不依赖 spike，不装额外 exe；单步/断点/连续都在浏览器里完成。 |

**为何不把 C 编译也放进浏览器（WASM）？**  
把完整 GCC/LLVM 编译到 WASM 体积大、集成复杂。用本地工具链一条命令即可，实现简单、行为与真实环境一致。

**为何模拟器用 JS 而不用 spike？**  
spike 需单独安装、需进程通信；JS 模拟器与前端同进程，断点/单步/刷新寄存器与内存视图最简单，且无额外依赖。

---

## 四、Windows 环境与安装（尽量少步骤）

1. **Node.js** 或 **Python 3**  
   - 官网下载安装包，勾选「Add to PATH」即可。

2. **RISC-V GNU 工具链（Windows 预编译）**  
   - 使用 xPack 或 SiFive 提供的 `riscv-none-elf-gcc` 等。  
   - 下载 zip，解压到固定目录（如 `C:\riscv-toolchain`），将该目录下的 `bin` 加入系统 PATH。  
   - 验证：命令行执行 `riscv-none-elf-gcc -v` 能输出版本即可。

3. **本项目**  
   - 拉取代码后，前端：用浏览器直接打开 `index.html`，或 `npx serve .` 提供静态页。  
   - 后端：`node server.js` 或 `python server.py`（一个文件即可），监听例如 `http://localhost:3000`。

**不采用**：Docker、WSL、MSYS2 编译工具链等，以降低在「纯 Windows」下的心智负担。

## 五、功能模块与实现要点

### 5.1 编辑与多视图

- **一个页面**内分栏或 Tab：
  - 输入：C 源码 / 带伪指令汇编（可切换或分两个输入框）。
  - 输出：展开后的汇编（无伪指令）、机器码（十六进制 + 反汇编行）。
- 实现：前端两个 `<textarea>` 或一个带语言切换的编辑器区域；「编译/汇编」按钮触发请求到后端，后端返回「汇编列表 + 机器码数组 + 反汇编文本」，前端渲染到对应区域。  
- **尽量简单**：可不做多文件，只做单文件编辑；文件读写可用「下载当前内容 / 从本地上传一个文件」实现，不做复杂工程结构。

### 5.2 代码转换链路（C / 带伪指令汇编 → 机器码）

- **C → 汇编（带伪指令）**  
  - 后端：把用户 C 源码写临时 `.c` 文件，执行  
    `riscv-none-elf-gcc -S -march=rv32im -mabi=ilp32 -O0 -fno-inline -o xxx.s xxx.c`  
  - 读生成的 `.s` 内容返回前端展示。

- **汇编（带伪指令）→ 汇编（无伪指令））**  
  - 方案 A：同一工具链的 `as` 汇编后，用 `objdump -d` 得到「展开后的反汇编」，把这部分文本当作「无伪指令汇编」展示（实现最简单）。  
  - 方案 B：若需「源码级」一一对应，可再研究 gas 的 `-a` 或 listing 输出（可选，非首版必须）。

- **汇编 → 机器码**  
  - 后端：`as -o xxx.o xxx.s`，再 `objcopy -O binary xxx.o xxx.bin` 或直接解析 ELF 的 .text 段，得到机器码字节数组；同时 `objdump -d xxx.o` 得到反汇编行。  
  - 前端：按「地址 + 机器码 + 反汇编」三列展示，并与「无伪指令汇编」视图对应（可通过地址或行号高亮）。

- **转换过程跟踪**  
  - 前端保留「当前是第几步」：例如 1=C→asm，2=asm→无伪指令，3=asm→机器码；每步的输入/输出在对应栏展示，并可选高亮当前步骤，逻辑简单。

### 5.3 模拟执行（单步 / 断点 / 连续）

- **模拟器**：用 JavaScript 实现一个 RV32IM 解释器（只实现用到的指令子集即可，先 RV32I+M）。
  - 数据结构：`regs[32]`、`pc`、`memory`（可 TypedArray 表示指令区 + 数据区，或统一线性空间分段）。
  - 指令：按 R/I/S/B/U/J 格式解析当前 `pc` 的 4 字节，查表执行对应语义；乘除法用 M 扩展 opcode。
- **执行模式**  
  - **单步**：执行一条指令，更新 `pc` 与 `regs`/内存，前端刷新寄存器表与内存视图。  
  - **断点**：用户在某地址或某行设断点；每执行一条前检查 `pc` 是否在断点集合，是则暂停并刷新 UI。  
  - **连续**：用 `requestAnimationFrame` 或 `setInterval` 循环单步，直到遇到断点或用户暂停或 `ebreak`/非法指令。
- **与「机器码」一致**：前端展示的机器码即模拟器加载的二进制；后端返回的 `bin` 或 ELF .text 原样交给模拟器，保证「编辑 → 汇编 → 机器码 → 执行」一致。

### 5.4 寄存器 / 指令存储器 / 数据存储器

- **寄存器**：模拟器内 `regs[0]..regs[31]` + `pc`；前端用表格展示，可读可写（初始化）；名称用 ABI 名（如 x1=ra, x2=sp）与数字索引双列。
- **指令存储器**：即加载的机器码对应区域；可按「地址 + 4 字节十六进制 + 反汇编」列表展示，支持从某地址开始执行。
- **数据存储器**：模拟器内划一段 Data 区（如 0x80000000 起）；支持按字节/半字/字查看与编辑（初始化），与课程要求的「初始化」对应。

实现细节尽量简单：内存用一段连续 ArrayBuffer，按地址分段区分 code/data；不实现 MMU、不实现真实 cache，只做平铺内存。

### 5.5 伪指令与基本指示性语句、系统调用

- **伪指令**：全部交给 GAS 处理（如 `li`, `la`, `mv`, `call`, `ret` 等），前端只展示「展开前 / 展开后」；不在前端再实现一遍展开逻辑。
- **指示性语句**：`.text`、`.data`、`.globl`、`.word`、`.asciz` 等由 as 支持；若需在编辑器中高亮，可做简单关键字高亮或接入轻量语法高亮库。
- **系统调用**：在模拟器内拦截 `ecall`；根据 `a7` 做少量语义（如 print_int, print_string, read_int, exit），数据从 `a0`/`a1` 等读取，结果写回 `a0`；不实现真实 Linux ABI，只做最小可用子集，便于实验。

## 六、后端 API 设计（极简）

- `POST /compile`  
  - Body: `{ "type": "c" | "asm", "source": "..." }`  
  - 返回：`{ "asm": "...", "asmExpanded": "...", "machineCode": [hex bytes], "disasm": "...", "errors": "..." }`  
  - 内部：写临时文件 → 调 gcc 或 as → 读 .s / .o / .bin，objdump 取反汇编；出错则返回 stderr。

- 可选：`GET /health`，用于检查本机是否已配好工具链（例如尝试执行 `riscv-none-elf-gcc -v` 并返回成功/失败）。

所有接口均 JSON，无登录、无持久化；临时文件用系统 temp 目录，执行完即删。


不引入数据库、不引入复杂构建（或仅保留一条 `npm run build` 可选）。

## 八、实现顺序建议

1. **环境**：在 Windows 上装好 Node 或 Python + RISC-V 工具链，能命令行完成 `gcc -S`、`as`、`objdump`。  
2. **后端**：实现 `POST /compile`（先只做 asm → 机器码），返回机器码 + 反汇编。  
3. **前端**：单页、输入汇编、点「汇编」后展示机器码与反汇编。  
4. **模拟器**：JS 实现最小 RV32I 子集（若干条常用指令），能加载机器码并单步执行，刷新寄存器。  
5. **扩展**：C 编译、伪指令展开展示、断点与连续执行、数据区与 ecall 系统调用。  
6. **完善**：寄存器/内存初始化、文件导入导出、错误提示与高亮。

---

## 九、风险与取舍

- **依赖本机工具链**：用户需自己安装 RISC-V GCC；通过文档 + `/health` 提示可降低排查成本。  
- **模拟器与真实 CPU 差异**：仅做教学演示足够；不保证与 spike 或硬件完全一致（如精确周期、异常模型）。  
- **C 标准与库**：仅支持单文件、无标准库或仅最小子集（如自己实现 putchar/getchar 的 ecall），避免链接 libc 的复杂性。

---

## 十、可行性结论

- **可行**：C/汇编编辑、多级代码展示、机器码生成全部依托成熟 RISC-V 工具链，后端只做调用与结果解析；模拟器用 JS 实现，无额外 exe，断点/单步/连续都好做。  
- **Windows 友好**：一次安装 Node（或 Python）+ 一个 RISC-V 工具链 zip 即可。  
- **实现简单**：核心是「一个后端 API + 一个前端页 + 一个 JS 模拟器」，先做最小闭环（汇编→机器码→单步执行），再逐步加 C、伪指令展示、断点与存储器视图。
